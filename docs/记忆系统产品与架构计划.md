# CareerOS 记忆系统产品与架构计划

## 1. 需求定义

CareerOS 的记忆系统不是简单的聊天记忆功能，而是个人职业成长操作系统的长期上下文层。它需要长期保存、组织、检索和复盘用户从学习到求职的完整成长轨迹。

核心需求分为三类：

1. 用户画像
2. 时间旅程
3. 学习文件和项目档案

### 1.1 用户画像

用户画像是稳定、短、每轮对话都可能需要的上下文。

典型内容：

- 学校、专业、年级、毕业时间
- 目标方向、目标岗位、目标公司类型、目标城市
- 技能、项目经历、实习经历、获奖经历
- 学习偏好、交互偏好、工具偏好
- 当前状态，例如正在学习什么、正在准备什么、焦虑程度、求职阶段

画像不应该存成长全过程，而应该是从长期事件中投影出来的当前快照。

### 1.2 时间旅程

时间旅程是用户成长过程的事件流。

典型内容：

- 用户什么时候上传了简历
- 用户什么时候决定学习某项技术
- 用户什么时候完成了某个项目
- 用户什么时候修改了职业目标
- 用户什么时候诊断过某个岗位
- 用户什么时候做过某次复盘
- 用户什么时候上传了学习笔记或项目档案

时间旅程适合用事件日志保存，而不是直接写进 Markdown。

### 1.3 学习文件和项目档案

用户未来会上传大量文件，包括：

- 简历
- 学习笔记
- 项目 README
- 课程作业
- 实习材料
- 面试复盘
- 作品集材料
- 技术调研文档

这些文件不适合完整写进 `memory.md`，也不应该只丢给 Cognee。系统必须保存原始文件和元数据，Cognee 只负责索引和召回。

## 2. 核心原则

记忆系统必须区分真相源、投影和索引。

```text
SQLite + 文件系统 = 真相源
.md 文件 = 可读快照投影
Cognee = 语义索引 / 图谱索引
```

原则：

- `growth_events` 记录长期成长事件。
- `documents` 记录上传文件元数据。
- 原始文件保存在文件系统。
- `.md` 文件从真相源生成，不作为主要写入源。
- Cognee 从真相源投影生成，不作为唯一长期记忆数据库。
- Cognee 可以删除、失败、重建，不影响真相源。
- Agent 每轮对话读取 `.md` 快照和相关长期召回结果。

## 3. 目标架构

```text
用户对话 / 手动编辑 / 简历上传 / 文件上传 / 项目档案
        |
        v
SQLite + 文件系统（真相源）
- growth_events
- documents
- projects
- raw files
        |
        +--> .md projector
        |       -> memory.md
        |       -> entities/*.md
        |
        +--> Cognee projector
                -> 语义索引
                -> 图谱/关系索引
```

### 3.1 真相源层

真相源负责保存事实，不负责 prompt 展示格式。

组成：

- `growth_events`：成长事件时间线。
- `documents`：上传文件元数据。
- `projects`：项目档案结构化数据，可选但建议后续增加。
- `raw files`：用户上传的原始文件。

### 3.2 投影层

投影层负责把真相源变成不同用途的读取形态。

组成：

- `.md projector`：生成 `memory.md` 和 `entities/*.md`。
- `Cognee projector`：生成语义索引和图谱索引。

### 3.3 使用层

使用层负责让 Agent 和前端消费记忆。

组成：

- `dynamic_prompt`：注入用户画像、长期召回、短期历史。
- `memory_search`：搜索 `.md`、events 和文档索引。
- `/api/memory/*`：管理事件、搜索、重建、清理。
- `/api/profile/*`：展示画像快照，不直接写 `.md`。
- 未来 `/api/documents/*`：管理上传文件和档案。
- 未来 `/api/timeline/*`：展示用户成长时间线。

## 4. 数据模型规划

### 4.1 `growth_events`

`growth_events` 是长期记忆的核心事件表。

字段建议：

```text
id
user_id
event_type
entity_type
entity_id
payload_json
source
dedupe_key
payload_hash
projected_md_at
projected_cognee_at
created_at
```

字段语义：

| 字段 | 说明 |
|---|---|
| `event_type` | 发生了什么，例如 `skill_added`、`goal_updated` |
| `entity_type` | 影响哪类对象，例如 `skill`、`goal`、`document` |
| `entity_id` | 影响哪个具体对象 |
| `payload_json` | 结构化事件内容 |
| `source` | 来源，例如 `conversation`、`resume`、`manual` |
| `dedupe_key` | 稳定去重键 |
| `payload_hash` | 内容哈希 |
| `projected_md_at` | 是否已投影到 Markdown |
| `projected_cognee_at` | 是否已投影到 Cognee 索引 |

常见事件类型：

```text
resume_uploaded
profile_updated
skill_added
skill_level_changed
experience_added
goal_updated
preference_learned
decision_made
status_changed
document_uploaded
document_indexed
project_created
project_updated
project_document_uploaded
learning_note_uploaded
journey_milestone_added
jd_diagnosed
memory_corrected
```

### 4.2 `documents`

`documents` 保存上传文件元数据。

建议字段：

```text
id
user_id
title
file_name
file_path
file_type
content_hash
source
summary
index_status
indexed_at
created_at
updated_at
```

字段语义：

| 字段 | 说明 |
|---|---|
| `file_path` | 原始文件在本地文件系统的位置 |
| `content_hash` | 文件内容哈希，用于去重 |
| `summary` | 文件摘要，可由 LLM 生成 |
| `index_status` | `pending`、`indexed`、`failed` |
| `indexed_at` | 投影到 Cognee 的时间 |

原始文件建议保存到：

```text
~/.careeros/files/
```

### 4.3 `document_chunks`

MVP 可以不建，后续如需自管切块再增加。

建议字段：

```text
id
document_id
chunk_index
chunk_text
content_hash
indexed_at
```

如果 Cognee 内部负责切块，系统可以先只保存 `documents` 元数据。

### 4.4 `projects`

项目档案可先用 `growth_events` 表达，后续产品化时建议独立成表。

建议字段：

```text
id
user_id
name
description
tech_stack
role
start_date
end_date
status
created_at
updated_at
```

项目相关文件通过 `documents` 关联：

```text
document.entity_type = "project"
document.entity_id = project_id
```

## 5. 记忆写入链路

### 5.1 标准写入链路

```text
用户行为
  -> 写 SQLite / 保存文件
  -> commit 成功
  -> 同步投影 .md 快照
  -> 异步投影 Cognee 索引
```

注意：

- `.md` 投影必须在 SQLite commit 成功之后执行。
- Cognee 投影可以异步执行。
- Cognee 投影失败不能回滚 SQLite。
- 失败事件需要可重试和可 rebuild。

### 5.2 对话写入

对话结束后自动提取长期记忆：

```text
user message + assistant reply
  -> memory_extractor
  -> structured events
  -> growth_events
  -> .md projector
  -> Cognee projector
```

适合提取：

- 新技能
- 新经历
- 新目标
- 新偏好
- 当前状态变化
- 重要决策

不适合提取：

- 闲聊
- 短期任务过程
- AI 自己推测但用户没有确认的事实
- 低置信度内容

### 5.3 简历上传

简历上传应事件化。

```text
resume file
  -> 保存原文件
  -> documents
  -> resume_uploaded event
  -> profile_updated events
  -> skill_added events
  -> experience_added events
  -> .md projector
  -> Cognee projector
```

MVP 可以先只写：

- `resume_uploaded`
- `profile_updated`

后续增强：

- 从简历结构化提取技能
- 从简历结构化提取项目和实习
- 将原始简历内容投影到 Cognee

### 5.4 文件上传

文件上传不直接写 `.md`，也不只写 Cognee。

```text
file upload
  -> 保存原始文件
  -> documents
  -> document_uploaded event
  -> 生成摘要
  -> Cognee 索引全文或切块
  -> .md 只记录摘要和引用
```

示例 `.md` 投影：

```text
## 项目档案
- FastAPI 简历分析器：document_id=xxx，摘要：实现了简历解析、JD 匹配和对话咨询。
```

## 6. 读取和召回链路

### 6.1 Agent prompt 注入

每轮对话建议注入：

```text
1. memory.md 核心画像
2. Cognee / SQLite 召回的相关长期记忆
3. 当前会话最近历史
```

默认不要把全部事件和全部文档塞进 prompt。

### 6.2 搜索策略

搜索应分层：

```text
精确搜索：
- SQLite events
- documents metadata
- .md 快照

语义搜索：
- Cognee
```

查询示例：

用户问：“我有哪些项目适合写进后端简历？”

系统应召回：

- `experience_added` 事件
- `project_created/project_updated` 事件
- 项目 README 文档
- 简历相关文档
- 当前目标岗位

## 7. 当前代码需要修正的点

当前实现已经开始向目标架构迁移，但仍有关键缺口。

### 7.1 修事务边界

当前风险：

```text
创建 event
-> flush
-> 写 .md
-> 调用方 commit
```

如果 commit 失败，`.md` 已经变化，但 `growth_events` 没有落库。

目标：

```text
创建 event
-> commit 成功
-> 写 .md
-> 后台投影 Cognee
```

### 7.2 修所有直接写 `.md` 的入口

需要改造：

- `/api/profile/me PATCH`
- `/api/profile/me DELETE`
- 旧 `update_profile` 工具
- 任何调用 `write_memory()` / `write_entity()` 的业务入口

原则：

- 用户主动编辑画像也应该写事件。
- `.md` 只能由 projector 写。

### 7.3 修 `dedupe_key`

当前问题：

- 如果没有 `entity_id`，同类事件容易互相误判重复。
- `payload_hash` 已有，但没有成为去重键的核心组成部分。

建议：

```text
dedupe_key = event_type + entity_type + entity_id + payload_hash
```

对部分实体可以使用稳定 key：

```text
skill: skill_name
goal: goal_name
document: content_hash
experience: title + period + payload_hash
```

### 7.4 修 Cognee 投影闭环

当前 Cognee 更多是 rebuild/recall 侧能力，不是实时索引。

需要：

- 新事件 commit 后后台投影 Cognee。
- 投影成功后写 `projected_cognee_at`。
- 投影失败记录错误，后续可重试。
- `/api/memory/rebuild` 从 SQLite 和文件系统重建 Cognee。

### 7.5 修 Cognee reset/rebuild 语义

当前 `clear_user_index()` 如果做不到真正清理，就不能假装已支持。

单用户模式可选方案：

- 删除本地 Cognee 存储目录后重新初始化。
- 或提供 `index_cleared=false`，明确告诉前端旧索引未清。

长期方案：

- 等 Cognee 支持按 metadata 删除。
- 或为每个用户维护独立 dataset。

### 7.6 补 SQLite 迁移

新增字段后需要给已有数据库补列：

- `dedupe_key`
- `payload_hash`
- `projected_md_at`
- `projected_cognee_at`

如果增加唯一约束，SQLite 需要特别处理，不能只靠 `ALTER TABLE ADD CONSTRAINT`。

## 8. 分阶段项目计划

### Phase 1：修稳事件真相源

目标：所有长期记忆写入都先进 `growth_events`。

任务：

- 修事务边界。
- 修直接写 `.md` 的入口。
- 修 dedupe key。
- 补 SQLite 迁移。
- 为 `create_event` 增加测试。

验收：

- 任意画像更新都能在 `growth_events` 看到。
- 删除 `memory.md` 后可以从 events 重建。
- 不存在业务入口直接覆盖 `memory.md`。

### Phase 2：完善 `.md` 投影器

目标：`.md` 成为稳定、可读、可重建的用户画像快照。

任务：

- 完善 profile/status/preferences/goals 合并规则。
- skills 按技能名或 `entity_id` 合并。
- experiences/decisions 保留时间线。
- `.md` 不保存文件全文，只保存摘要和引用。
- 投影使用原子写入。

验收：

- 重复投影结果一致。
- 多次技能更新只展示最新状态。
- 经历和决策不会被覆盖丢失。

### Phase 3：Cognee 索引化

目标：Cognee 成为可重建、可丢弃、可重试的索引层。

任务：

- 新事件 commit 后异步投影 Cognee。
- 成功后写 `projected_cognee_at`。
- 失败后保留待重试状态。
- `/api/memory/rebuild` 可重建 Cognee。
- 单用户模式下实现真正清空 Cognee 索引，或明确返回不支持。

验收：

- 新增事件后 Cognee 能召回。
- Cognee 删除后可以从 SQLite 重建。
- reset 后不会召回旧记忆。

### Phase 4：文件和项目档案

目标：支持学习文件、项目档案、简历材料等长期资料。

任务：

- 新增 `documents` 表。
- 保存原始文件到 `~/.careeros/files/`。
- 写入 `document_uploaded` 事件。
- 生成文件摘要。
- 将文件内容投影到 Cognee。
- `.md` 只显示摘要和引用。

验收：

- 上传文件后能在 documents 看到。
- 原始文件可定位。
- 用户可以问“我之前上传过什么项目资料”。
- 用户可以问“根据我的项目档案帮我改简历”。

### Phase 5：时间旅程产品化

目标：把事件流变成用户可看的成长路径。

任务：

- 新增 `/api/memory/timeline`。
- 支持按类型筛选事件。
- 前端做成长时间线页面。
- 支持阶段总结，例如周总结、月总结、求职阶段复盘。

验收：

- 用户能看到自己的成长轨迹。
- AI 能基于时间线做复盘。
- 文件、项目、技能、目标能串起来。

## 9. 产品场景示例

### 9.1 项目档案上传

```text
用户上传 FastAPI 项目 README
  -> 保存原文件
  -> documents 写入元数据
  -> growth_events 写 project_document_uploaded
  -> Cognee 索引 README 内容
  -> .md 记录项目摘要和 document_id
```

之后用户问：

```text
我有哪些项目适合写进后端简历？
```

系统可以召回：

- 项目 README
- 项目相关事件
- 用户当前目标岗位
- 用户已有技能

### 9.2 学习旅程复盘

用户问：

```text
帮我总结一下这个月的学习进展。
```

系统可以读取：

- 本月 `growth_events`
- 学习笔记 documents
- 技能变化事件
- 当前目标变化

然后生成阶段复盘。

### 9.3 JD 诊断延续

用户上传 JD 或让系统分析岗位。

系统写入：

```text
jd_diagnosed
skill_gap_found
goal_updated
```

后续用户问：

```text
我之前后端岗位诊断缺什么？
```

系统可以从 events 和 Cognee 召回历史诊断。

## 10. 最终判断

推荐架构是：

```text
growth_events / documents / raw files 是事实。
.md 是画像快照。
Cognee 是语义索引。
```

这套架构适合 CareerOS，因为它同时满足：

- 每轮对话需要稳定画像。
- 长期成长需要时间线。
- 文件和项目档案需要原始资料保存。
- AI 需要语义召回。
- 索引层需要可重建、可丢弃。

短期重点不是做复杂图谱，而是把真相源、投影、索引三层边界打牢。
