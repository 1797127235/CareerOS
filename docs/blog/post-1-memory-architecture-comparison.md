# 新手用两个月搭了一个 AI 伴侣：我对比了 7 种记忆方案，最后选择事件溯源

> 这是我的第一个 AI 项目。如果你也在入门 AI Agent 开发，希望我的踩坑记录对你有用。
> 
> 项目开源：[github.com/1797127235/Lumen](https://github.com/1797127235/Lumen)

---

## 为什么要做这个项目

我去年开始学 AI 开发，看了很多教程，发现一个共同的问题：**示例代码里的 Agent 都是"金鱼记忆"**——每次对话结束，之前聊的内容就全丢了。

真实的 AI 伴侣（Companion AI）应该像人一样，随着时间了解你：

- 你提过下周有个面试
- 你说过自己在学 TypeScript
- 你分享过对远程工作的看法

下一次聊天，它应该记得这些，而不是问"你是做什么工作的？"

于是我决定动手做一个真正能记住用户的 AI 伴侣。技术栈选了 FastAPI + PydanticAI + React，数据库用 SQLite（自托管优先，零运维成本）。

做到记忆系统时，我发现这是一个**比想象中复杂得多的问题**。

---

## AI 记忆系统的核心矛盾

大语言模型有两个硬约束：

1. **上下文窗口有限**（通常 4K-128K tokens）
2. **每次对话都是无状态的**（模型本身不保留跨会话的记忆）

所以所有 AI 伴侣都要回答同一个问题：**用户的一生可能有几百万字的信息，怎么塞进几千字的上下文窗口？**

我花了一周调研现有的开源方案，整理出 7 种代表性架构：

---

## 7 种 AI 记忆方案对比

| 方案 | 核心思想 | 记忆层次 | 复杂度 |
|------|---------|---------|--------|
| **Letta (MemGPT)** | OS 虚拟内存类比：RAM ↔ Disk 分页 | 3 层：Context → Recall → Archival | 高 |
| **Soul Protocol** | 心理学 ACT-R 理论 | 5 层：Episodic → Semantic → Procedural → Context → Archival | 很高 |
| **Cortex** | 事件溯源 + 投影简报 | 存储层(events) + 投影层(briefings) | 中 |
| **FadeMem** | 生物遗忘曲线 | 双层：Working → Long-term | 低 |
| **Mio** | 对话感知记忆 | Episodic + Semantic | 低 |
| **Memory as Metabolism** | 知识新陈代谢 | 3 层：交互/表征/保留 | 中 |
| **StatePlane** | 认知状态平面 | Episodic + Semantic + Procedural | 中 |

### 方案 1：Letta（前身 MemGPT）

Letta 的思路最直观——把上下文窗口当成 RAM，把外部存储当成 Disk。

当 RAM 满了，就把不常用的内容"换出"到 Disk；需要的时候再"换入"。这个分页机制由 Agent 自己决定，通过 `core_memory_replace`、`archival_memory_search` 等工具操作。

**优点**：概念清晰，和操作系统类比容易理解  
**缺点**：实现复杂，需要维护两套存储的同步状态；分页决策本身也会消耗上下文

### 方案 2：Soul Protocol

Soul Protocol 基于认知心理学 ACT-R 理论，把记忆分成 5 层：

- **Episodic**：具体事件（"我上周面试了字节"）
- **Semantic**：抽象知识（"字节跳动是一家互联网公司"）
- **Procedural**：程序性知识（"面试时要准备 STAR 法则"）
- **Context**：当前对话上下文
- **Archival**：冷存储

**优点**：理论完备，分层合理  
**缺点**：5 层之间的流转规则复杂，新手难以落地

### 方案 3：Cortex ⭐

Cortex 是我最终参考最多的方案。它的核心设计是**事件溯源 + 投影简报**：

```
存储层（Events）    ← 所有事实以不可变事件存储
    ↓
投影层（Projections）← 事件聚合为可读视图
    ↓
简报层（Briefings）  ← 投影进一步压缩为决策级摘要
```

**关键创新**：不是把原始事件注入 Prompt，而是注入压缩后的简报。简报默认只占约 2,000 tokens（~0.1% 上下文窗口）。

这个设计的核心思想是：**分离 retention（存储一切）和 representation（只展示重要内容）**。

### 方案 4：Mio

Mio 是另一个 AI 伴侣项目的记忆策略，最打动我的是它的**类型特异性衰减**：

| 记忆类型 | 衰减速度 | 原因 |
|---------|---------|------|
| 基本资料（profile）| 不衰减 | 身份是稳定的 |
| 目标（goals）| 月级 | 目标变化慢 |
| 技能（skills）| 周级 | 技能会更新 |
| 经历（experiences）| 日级 | 经历很快过时 |
| 对话碎片（chat snippets）| 会话结束即衰减 | 临时信息 |

**优点**：直觉合理，不需要复杂算法  
**缺点**：没有考虑"重要度"差异（一次重要决策 vs. 一次闲聊碎片）

### 方案 5：FadeMem

FadeMem 用艾宾浩斯遗忘曲线给记忆打分，越久不用的记忆权重越低。

**优点**：拟人化，有生物学依据  
**缺点**：衰减曲线参数难调；重要但久未提及的信息可能被打低分

### 另外两种

- **Memory as Metabolism**：把知识比作新陈代谢，有摄入、消化、排泄三个阶段。概念有趣但实现模糊。
- **StatePlane**：把记忆组织成"认知平面"，强调不同记忆类型之间的关联。偏学术研究，工程落地参考性一般。

## 我的选择：混合轻量方案

作为新手，我没有选择任何一个纯方案，而是取了各家之长，做了一个**不改存储、只改注入**的混合方案：

### 核心思想

**Cortex 的简报层 + Mio 的类型衰减 + Soul Protocol 的排序思路**

### 实现：三层注入

```python
async def build_snapshot(user_id: str, user_input: str) -> str:
    # L0: 固定块（永远注入，< 800 字）
    # 核心画像 + 当前目标 + 关键技能
    fixed = build_fixed_block(user_id)
    
    # L1: 近期块（最近 10 条事件，应用类型衰减）
    recent = get_recent_events(user_id, limit=10)
    recent = apply_decay_filter(recent)  # 按事件类型加权
    
    # L2: 语义召回（与用户当前输入相关）
    relevant = semantic_recall(user_id, user_input, limit=5)
    
    return merge_and_deduplicate(fixed, recent, relevant)
```

### 类型衰减权重

我定义了简单的衰减系数：

| 事件类型 | 衰减系数 | 说明 |
|---------|---------|------|
| `profile_updated` | 0.0 | 身份信息，永不衰减 |
| `goal_updated` | 0.1 | 目标变化慢 |
| `skill_added` | 0.2 | 技能周级更新 |
| `preference_learned` | 0.3 | 偏好中等衰减 |
| `experience_added` | 0.5 | 经历日级衰减 |
| `chat_fragment` | 1.0 | 不注入 L1，仅语义召回 |

具体实现用 `衰减系数 × 时间衰减（30 天线性到 0）` 计算最终权重，低于阈值（0.2）的过滤掉。

### 简报层怎么压缩

这是最关键的部分。我不直接注入全量的 `.md` 投影文件（可能有 4000-5000 字），而是预生成一个压缩的 `briefing`：

```
【核心画像】（~300 字）
  姓名、身份、当前阶段、核心诉求

【活跃目标】（~200 字）
  正在追求的 2-3 个目标

【关键技能】（~200 字）
  Top 5 技能

【近期动态】（~500 字）
  最近 10 条过滤后的事件

【相关记忆】（~300 字）
  与用户当前输入语义相关的 5 条
```

总 Prompt 大小从原来的 **10,000+ tokens** 降到 **1,500-2,500 tokens**，信息密度反而更高。

## 实现中的三个关键决策

### 1. 为什么用 SQLite 而不是向量数据库做主存储？

我试过 Pinecone、Chroma、LanceDB，最后选了 SQLite + FTS5 做主存储，Cognee 做语义增强。

原因很简单：**这个项目定位是自托管**。要求用户额外跑一个向量数据库服务，门槛太高了。SQLite 是单文件，零配置，备份就是一个 `cp` 命令。

向量检索用 Cognee（也是本地跑），还没完全调通，所以目前用 FTS5 全文搜索兜底。

### 2. 为什么让 Agent 主动调用工具保存记忆，而不是后台自动提取？

早期的做法是对话结束后用一个后台任务自动分析内容、提取记忆。但有两个问题：

1. **不准确**：自动提取经常误判，把随口一说当成重要信息
2. **不可控**：用户不知道什么被记住了，缺乏透明度

改成 Agent 工具驱动后，Agent 在对话中主动判断"这条信息值得保存"，然后调用 `memory_save`。透明度更高，准确率也更好。

当然，Agent 有时候会漏掉。所以加了一个**后台审查兜底**：如果一轮对话 Agent 没有调用记忆工具，就 fork 一个独立 Agent 审查对话内容，看是否有遗漏。

### 3. 为什么不用现成的框架，从头搭？

说实话，Letta、Dify、Coze 这些平台都能快速搭出带记忆的 Agent。但我做这个项目的目的不是"搭一个产品"，而是**理解底层原理**。

自己实现事件溯源、投影器、工具调用循环、流式 SSE——这些过程让我真正理解了 Agent 是怎么工作的。如果你也在学 AI 开发，我强烈建议至少亲手搭一次。

## 给同样在入门的朋友

如果你也在考虑做第一个 AI Agent 项目，我的建议是：

1. **从具体场景切入**：不要做一个"通用 Agent"，选一个你真正关心的场景。我选"职业规划伴侣"是因为我自己就在这个阶段。
2. **不要过度设计**：我一开始也想做 5 层记忆架构，后来发现 3 层够用，代码简单才能持续迭代。
3. **公开记录过程**：我写这篇博客的时候，发现很多"理所当然"的设计其实都经历过纠结。记录这些纠结，对后来者很有价值。
