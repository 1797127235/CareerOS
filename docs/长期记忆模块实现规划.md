# CareerOS 长期记忆模块实现规划

## 1. 定位

本项目长期记忆主线确定为：

```text
growth_events SQLite 真相源
  -> cognee_projector 投影
  -> Cognee 语义 / 图谱检索
  -> /api/memory/* 管理
  -> Agent dynamic_prompt 注入
```

`memory.md + entities/*.md` 不作为长期记忆主线，而作为画像记忆 / 用户档案层使用。

三类记忆边界如下：

| 层级 | 名称 | 存储 | 职责 |
|---|---|---|---|
| L1 | 短期对话记忆 | `conversations` / `messages` / `conversation.summary` | 当前会话连续性、最近上下文 |
| L2 | 画像记忆 | `~/.careeros/memory.md` 与 `entities/*.md` | 用户稳定画像：学校、专业、技能、经历、偏好、目标 |
| L3 | 长期事件记忆 | `growth_events` + Cognee + `/api/memory/*` | 跨会话事件、成长轨迹、目标变化、决策、JD 诊断历史 |

核心原则：

- `growth_events` 是长期记忆真相源。
- Cognee 是长期记忆索引层和语义检索层。
- `/api/memory/*` 管理长期事件记忆，不管理画像 md 文件。
- Agent 每轮对话可以读取画像记忆，也应该召回相关长期事件记忆。

## 2. 当前代码状态

已存在的模块：

| 文件 | 当前职责 | 状态 |
|---|---|---|
| `app/backend/models/growth_event.py` | 定义 `GrowthEvent` 表 | 已有，但写入点不足 |
| `app/backend/services/cognee_projector.py` | 把 `growth_events` 投影到 Cognee | 已有，但未接入主流程 |
| `app/backend/services/cognee_service.py` | `remember/recall/forget/rebuild` 封装 | 已有，但 `recall()` 当前主要走 SQLite 降级 |
| `app/backend/routers/memory.py` | `/api/memory/stats/list/reset` | 已有，但接口语义需要收紧为长期事件记忆 |
| `app/backend/services/chat_service.py` | 对话主流程 | 未做对话后长期记忆提取 |
| `app/backend/services/profile_service.py` | 简历解析与画像写入 | 目前主要写 `memory.md`，未产出 `growth_events` |
| `app/backend/services/skill_service.py` | 技能 CRUD | 未产出 `growth_events` |
| `app/backend/services/jd_service.py` | JD 诊断 | 未产出 `growth_events` |
| `app/backend/agent/pydantic_agent.py` | Agent prompt 注入 | 主要注入画像 md 与短期历史，未注入长期事件召回 |

当前缺口：

- 业务动作没有统一写入 `GrowthEvent`。
- `cognee_projector.project_event()` 没有被写入流程调用。
- 对话结束后没有自动提取长期记忆。
- `cognee_service.recall()` 还没有真正形成 Cognee 优先、SQLite 降级的两阶段检索。
- `/api/memory/reset` 会删除 `growth_events`，同时 `cognee_service.forget()` 内部也会删除一次，存在职责重复。

## 3. 目标架构

### 3.1 写入链路

所有长期记忆写入统一走事件服务：

```text
业务动作
  -> growth_event_service.create_event(...)
  -> SQLite 写入 growth_events
  -> fire-and-forget cognee_projector.project_event(event)
  -> Cognee remember
```

需要新增：

```text
app/backend/services/growth_event_service.py
```

建议提供这些函数：

```python
async def create_growth_event(
    db,
    user_id: str,
    event_type: str,
    entity_type: str | None = None,
    entity_id: str | None = None,
    payload: dict | None = None,
    source: str = "system",
    project: bool = True,
) -> GrowthEvent:
    ...

async def list_growth_events(db, user_id: str, limit: int = 100) -> list[GrowthEvent]:
    ...

async def delete_growth_events(db, user_id: str) -> int:
    ...
```

投影失败不回滚 SQLite。长期记忆真相源必须优先保证落库。

### 3.2 读取链路

Agent 每轮对话上下文由三部分组成：

```text
画像记忆摘要
  -> read_memory()

长期事件召回
  -> cognee_service.recall(user_id, user_input, limit=5)

短期对话历史
  -> messages 最近 N 条 / conversation.summary
```

推荐顺序：

```text
用户画像
长期相关记忆
当前会话历史
```

长期事件召回应该短而准。默认注入 3-5 条即可，避免污染 prompt。

## 4. GrowthEvent 事件模型

当前 `GrowthEvent` 字段基本够用：

| 字段 | 用途 |
|---|---|
| `user_id` | 用户隔离 |
| `event_type` | 发生了什么 |
| `entity_type` | 影响哪类对象 |
| `entity_id` | 影响哪个对象 |
| `payload_json` | 事件详情 |
| `source` | 事件来源 |
| `created_at` | 时间线 |

建议后续加字段：

| 字段 | 类型 | 原因 |
|---|---|---|
| `conversation_id` | `String(36), nullable=True` | 追溯事件来自哪轮对话 |
| `confidence` | `Float, default=1.0` | 区分用户明确表达和模型推断 |
| `projected_at` | `DateTime, nullable=True` | 记录是否已投影 Cognee |
| `projection_status` | `String(20), default="pending"` | 支持重试和管理页展示 |

MVP 可以先不迁移字段，先用现有字段打通链路。

## 5. 事件类型规范

第一版收敛到这些事件类型：

| event_type | entity_type | 来源 | 说明 |
|---|---|---|---|
| `profile_updated` | `profile` | 简历上传 / 画像编辑 / 工具调用 | 用户画像发生变化 |
| `resume_uploaded` | `profile` | 简历上传 | 上传并解析简历 |
| `skill_added` | `skill` | 技能 CRUD / 对话提取 | 新增技能 |
| `skill_level_changed` | `skill` | 技能更新 / 对话提取 | 技能熟练度变化 |
| `experience_added` | `experience` | 简历上传 / 对话提取 | 新增项目、实习、竞赛经历 |
| `goal_updated` | `goal` | 对话提取 / 画像编辑 | 目标方向变化 |
| `preference_learned` | `preference` | 对话提取 | 学习、交互、岗位偏好 |
| `decision_made` | `decision` | 对话提取 | 用户做出重要决策 |
| `status_changed` | `status` | 对话提取 | 求职、学习、心理状态变化 |
| `jd_diagnosed` | `jd` | JD 诊断 | 完成一次岗位匹配分析 |
| `memory_corrected` | `memory` | 用户纠错 | 用户否定或修正已有记忆 |

`payload_json` 只存结构化事实，不存长篇 AI 回复。

示例：

```json
{
  "skill_name": "FastAPI",
  "level": "familiar",
  "evidence": "用户说自己做过 FastAPI 项目",
  "confidence": 0.8
}
```

## 6. 写入点规划

### 6.1 简历上传

位置：

```text
app/backend/services/profile_service.py
```

成功解析并写入画像后，写入：

- `resume_uploaded`
- `profile_updated`
- `skill_added`，按解析出的技能逐条写
- `experience_added`，按项目 / 实习 / 奖项逐条写

注意：

- 简历上传是高置信来源，`source="resume"`。
- 如果第一版 LLM 只返回 markdown，不返回结构化 JSON，则先写 `resume_uploaded` 和 `profile_updated`，后续再改解析输出。

### 6.2 技能 CRUD

位置：

```text
app/backend/services/skill_service.py
```

写入：

- `create_skill()` 成功后写 `skill_added`
- `update_skill()` 熟练度变化后写 `skill_level_changed`
- 删除技能是否写事件要谨慎。第一版可以不写，或写 `memory_corrected`

### 6.3 JD 诊断

位置：

```text
app/backend/services/jd_service.py
```

诊断成功后写：

- `jd_diagnosed`

payload 建议包含：

```json
{
  "jd_title": "...",
  "overall_score": 76,
  "matched_skills": ["Python", "FastAPI"],
  "skill_gaps": ["系统设计", "工程化经验"],
  "summary": "..."
}
```

这类记忆适合长期召回，因为它反映用户曾经评估过什么岗位、缺口是什么。

### 6.4 对话后自动提取

位置：

```text
app/backend/services/chat_service.py
```

在 AI 回复保存成功后启动后台任务：

```python
task = asyncio.create_task(
    _extract_long_term_memory_bg(
        user_id=user_id,
        conversation_id=conv.conversation_id,
        user_input=user_input,
        assistant_reply=full_content,
    )

task.add_done_callback(_log_task_error)
```

新增：

```text
app/backend/services/memory_extractor.py
```

职责：

1. 判断本轮是否包含值得长期记忆的信息。
2. 调用 LLM 输出结构化事件列表。
3. 过滤低置信度、空事件、重复事件。
4. 调用 `growth_event_service.create_growth_event()` 写入。

提取 prompt 要求输出 JSON：

```json
{
  "events": [
    {
      "event_type": "goal_updated",
      "entity_type": "goal",
      "entity_id": "career_direction",
      "payload": {
        "goal": "AI Agent 开发",
        "reason": "用户明确表达目标方向"
      },
      "source": "conversation",
      "confidence": 0.9
    }
  ]
}
```

第一版只保存 `confidence >= 0.7` 的事件。

### 6.5 Agent 工具调用

位置：

```text
app/backend/agent/pydantic_tools.py
```

当前 `memory_update` 和 `memory_add` 主要操作 md 画像记忆。建议拆分语义：

- `profile_memory_update`：更新 `memory.md/entities/*.md`
- `long_term_memory_add`：写 `growth_events`

MVP 可以先保留现有工具，但在工具成功后补一条 `GrowthEvent`。

示例：

- `memory_update(entity_type="memory")` 成功后写 `profile_updated`
- `memory_add(entity_type="goals")` 成功后写 `goal_updated`
- `memory_add(entity_type="decisions")` 成功后写 `decision_made`

## 7. Cognee 投影规划

### 7.1 投影职责

`cognee_projector.project_event(event)` 保持单向投影：

```text
GrowthEvent -> human readable memory content -> cognee_service.remember()
```

投影内容要带 `user_id`，避免 Cognee 元数据无法区分用户。

当前 `cognee_service._cognee_metadata()` 只合并 dataset，建议 metadata 永远包含：

```json
{
  "dataset": "career_os",
  "user_id": "...",
  "event_id": "...",
  "event_type": "...",
  "entity_type": "...",
  "entity_id": "...",
  "source": "..."
}
```

### 7.2 投影失败处理

第一版：

- 写 SQLite 成功即视为长期记忆写入成功。
- Cognee 投影失败只打日志。
- `/api/memory/rebuild` 可从 `growth_events` 重建索引。

第二版：

- 在 `growth_events` 加 `projection_status/projected_at`。
- 后台重试 pending/failed 事件。

## 8. Recall 规划

`cognee_service.recall()` 改成两阶段：

```text
1. Cognee 语义召回，带 user_id metadata 过滤
2. 失败或结果为空时，从 SQLite growth_events 降级召回
```

注意：

- 如果 Cognee 当前无法可靠按 `user_id` 过滤，就不要直接全局语义搜索。
- 可以先保留 SQLite recall 作为默认实现，同时把投影链路打通。
- 等确认 Cognee metadata filter 可用后，再启用 Cognee 优先。

SQLite 降级第一版可以按时间倒序，第二版再做关键词过滤：

```text
query 命中 payload_json / event_type / entity_type
```

## 9. API 规划

现有：

```text
GET  /api/memory/stats
GET  /api/memory/list
POST /api/memory/reset
```

建议语义固定为长期事件记忆：

| Endpoint | 职责 |
|---|---|
| `GET /api/memory/stats` | 返回 growth_events 数量、Cognee 状态、投影状态 |
| `GET /api/memory/list` | 返回 growth_events 时间线 |
| `POST /api/memory/reset` | 删除该用户 growth_events，并清理或重建 Cognee |
| `POST /api/memory/rebuild` | 从 growth_events 重建 Cognee |
| `DELETE /api/memory/{event_id}` | 删除单条长期记忆事件 |

第一阶段只需要补 `rebuild`，单条删除可以后置。

`reset_memory()` 当前同时在 router 中删除 SQLite，又调用 `cognee_service.forget()`，而 `forget()` 内部也删除 SQLite。建议改成：

```text
router 删除 growth_events
router 调用 cognee_service.clear_user_index(user_id)
```

不要让 Cognee 服务删除 SQLite 真相源。

## 11. 实施阶段

### Phase 1：打通 SQLite 真相源

目标：

- 所有关键动作写入 `growth_events`。
- `/api/memory/list/stats/reset` 基于 `growth_events` 稳定工作。

任务：

- 新增 `growth_event_service.py`。
- 简历上传写 `resume_uploaded/profile_updated`。
- 技能新增 / 更新写事件。
- JD 诊断写事件。
- 修正 `/api/memory/reset` 的职责重复。
- 增加基础测试。

验收：

- 上传简历后 `/api/memory/list` 能看到事件。
- 新增技能后 `/api/memory/list` 能看到 `skill_added`。
- 重置后 `growth_events` 清空。

### Phase 2：接入 Cognee 投影

目标：

- 新事件写入后异步投影到 Cognee。
- 支持从 SQLite 重建 Cognee。

任务：

- `create_growth_event(..., project=True)` 内部触发 `project_event()`。
- `/api/memory/rebuild` 调用 `cognee_projector.project_all_events(user_id)`。
- `cognee_service.remember()` metadata 增加 `user_id/event_id`。

验收：

- 写事件不会因 Cognee 失败而失败。
- rebuild 能重放该用户所有事件。

### Phase 3：对话后自动提取

目标：

- 用户在对话中表达目标、偏好、决策、经历后，系统自动沉淀长期记忆。

任务：

- 新增 `memory_extractor.py`。
- 在 `chat_service.py` 保存 AI 回复后启动后台提取。
- 提取结果写 `growth_events`。
- 增加重复过滤。

验收：

- 用户说“我决定主攻 AI Agent 开发”，对话结束后生成 `decision_made` 或 `goal_updated`。
- 用户说“我做过一个 FastAPI 项目”，生成 `experience_added` 或 `skill_added`。

### Phase 4：长期记忆召回注入

目标：

- Agent 回答时能利用跨会话长期事件。

任务：

- `CareerOSDeps` 增加 `current_user_input`。
- `dynamic_prompt` 调用 `cognee_service.recall()`。
- 限制注入条数和长度。

验收：

- 新会话中用户问“结合我之前的经历给建议”，Agent 能引用长期记忆事件。

### Phase 5：Cognee 优先语义检索

目标：

- `recall()` 先用 Cognee 召回，失败降级 SQLite。

任务：

- 验证 Cognee metadata 过滤能力。
- 实现 Cognee query。
- 保留 SQLite fallback。

验收：

- 语义相近但关键词不同的问题能召回历史事件。
- 不会跨用户泄露记忆。

## 12. 测试规划

第一批测试应覆盖：

- `create_growth_event()` 能写入 payload JSON。
- `memory/stats` 返回指定用户事件数。
- `memory/list` 只返回指定用户事件。
- `memory/reset` 只删除指定用户事件。
- Cognee 不可用时，写入事件不失败。
- `cognee_projector._build_memory_content()` 对主要事件类型输出可读文本。

后续集成测试：

- 简历上传产生长期记忆事件。
- 技能新增产生长期记忆事件。
- JD 诊断产生长期记忆事件。
- 对话后提取产生长期记忆事件。

## 13. 风险与约束

### Cognee 用户隔离

如果 Cognee 不能稳定按 metadata 过滤 `user_id`，不能直接启用全局语义检索。必须继续使用 SQLite recall 或用户级 dataset。

### 事件重复

对话后提取容易反复写入同一事实。第一版可以用简单规则过滤：

```text
同 user_id + event_type + entity_type + entity_id + payload 核心字段
在最近 N 天已存在，则跳过
```

### Prompt 污染

长期记忆召回不应一次注入太多。默认 3-5 条，单条截断到 200 字以内。

### 画像记忆和长期事件记忆混淆

`memory.md` 不再承担长期事件记忆职责。它只提供用户档案摘要。长期事件进入 `growth_events`。

## 14. 推荐优先级

建议立即做：

1. 新增 `growth_event_service.py`。
2. 简历上传、技能 CRUD、JD 诊断写入 `growth_events`。
3. 修正 `/api/memory/*` 明确管理 `growth_events`。
4. 接入 `cognee_projector.project_event()`。
5. 再做对话后自动提取。

这个顺序能先把长期记忆真相源跑通，再逐步增强 Cognee 和 Agent 召回。
