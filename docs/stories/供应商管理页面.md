## 状态管理

openhanako 用 Zustand `useSettingsStore`，管理：
- `providersSummary: Record<string, ProviderSummary>` — 所有供应商状态
- `selectedProviderId: string | null` — 当前选中供应商
- `showToast(msg, type)` — 1.5 秒 toast

Lumen 新建 `src/store/providersStore.ts`，用 Zustand，照抄 openhanako `store.ts` 里 providers 相关的字段和 `showToast` action，去掉所有无关字段（agents、serverPort 等）。

```typescript
// src/store/providersStore.ts
import { create } from 'zustand'

interface ProvidersStore {
  providersSummary: Record<string, ProviderSummary>
  selectedProviderId: string | null
  toastMessage: string
  toastType: 'success' | 'error' | ''
  toastVisible: boolean
  showToast: (message: string, type: 'success' | 'error') => void
}
```

如果项目还没有 zustand，运行 `npm install zustand`。

## 数据结构

### ProviderSummary（照抄 openhanako store.ts）

```typescript
interface ProviderSummary {
  type: 'api-key'
  auth_type: 'api-key' | 'none'
  display_name: string
  base_url: string
  api: string           // 'openai-completions' | 'anthropic-messages' | 'google-generative-ai'
  api_key: string
  models: (string | { id: string; name?: string; context?: number; image?: boolean; video?: boolean; reasoning?: boolean })[]
  custom_models: string[]
  has_credentials: boolean
  supports_oauth: false
  can_delete: boolean
  config_status: 'ok' | 'needs_setup' | 'invalid'
  config_error: string | null
  missing_fields: string[]
}
```

## 后端接口（照抄 openhanako providers.js 的逻辑，用 FastAPI 实现）

### GET /api/providers/summary

数据来源两层合并：
1. `PROVIDER_CATALOG`（`backend/core/config.py`）— 预置供应商（openai/anthropic 等），`can_delete: False`
2. `config.json["providers"]` — 用户配置（api_key/base_url/models），覆盖预置数据
3. `config.json["providers"]` 里有但不在 `PROVIDER_CATALOG` 的 key — 用户自建供应商，`can_delete: True`

每条 summary 逻辑（照搬 providers.js 第 95-128 行）：
- `has_credentials` = api_key 非空
- `missing_fields` = 缺少 api_key 时包含 `"api_key"`，models 为空时包含 `"models"`
- `config_status` = missing_fields 为空 → `"ok"`，否则 → `"needs_setup"`

### POST /api/providers/fetch-models

body: `{ name, base_url?, api?, api_key? }`

凭证优先级：body > config.json > 空

按 `api` 字段调不同端点（照搬 providers.js 第 288-349 行）：
- `openai-completions` → `GET {base_url}/models`，Header: `Authorization: Bearer {api_key}`
- `anthropic-messages` → `GET {base_url}/v1/models?limit=1000`，Header: `x-api-key: {api_key}`
- `google-generative-ai` → `GET https://generativelanguage.googleapis.com/v1beta/models?key={api_key}`

成功后写 `~/.lumen/models-cache.json`，返回 `{ models: [...] }`。

fallback：远程失败时，从 `PROVIDER_CATALOG[name]["chat_models"]` 返回预置列表。

### GET /api/providers/{name}/discovered-models

读 `~/.lumen/models-cache.json`，返回 `{ models: [...], fetchedAt: "..." }`。

### POST /api/providers/test

body: `{ name?, base_url, api, api_key? }`

凭证优先级同 fetch-models。

调对应端点（同上），收到任何 2xx 返回 `{ ok: true }`，收到 401/403 或异常返回 `{ ok: false, error: "..." }`。

### PUT /api/providers/{name}/models/{model_id}

body: `{ name?, context?, maxOutput?, image?, video?, reasoning? }`

将这些字段合并写入 `config.json["providers"][name]["models"]` 里对应 model_id 的对象。
如果该 model 当前是字符串形式（`"gpt-4o"`），转换为对象形式（`{ "id": "gpt-4o", ... }`）再写入。

### PUT /api/config（现有接口）

openhanako 用 `PUT /api/config` 保存 api_key / base_url / models 列表（`{ providers: { openai: { ... } } }`）。

Lumen 现有 `backend/modules/config/router.py` 已有 `PUT /api/config`，检查它是否支持保存嵌套的 `providers` 字段到 config.json。如果不支持，扩展它。
