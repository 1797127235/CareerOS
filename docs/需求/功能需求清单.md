## 三、功能需求清单

按 **MoSCoW 法则** 分类，以下为系统核心功能需求：

---

### Must Have（必须有） — 产品最小可用版本（MVP）的核心功能

| 需求编号 | 需求名称 | 需求描述 | 优先级理由 |
|----------|----------|----------|------------|
| M-01 | 智能对话式职业咨询 | 用户可通过自然语言与AI智能体进行多轮对话，询问计算机行业各方向的信息、职业发展路径、学习建议等。智能体基于大模型知识库给出结构化、个性化的回答。 | 这是系统的核心交互形态，直接解决用户"迷茫、信息不对称"的最大痛点。如果没有此功能，产品失去AI Agent的核心价值。 |
| M-02 | 个性化学习路径生成 | 系统根据用户的目标岗位、当前基础、可用时间，生成定制化的技术学习路径，按阶段拆解学习任务，标注学习资源（书籍/视频/文档）和验收标准。 | 用户画像2（焦虑的追赶者）的核心诉求。调研显示"学习路径碎片化"是学生最大痛点之一，此功能是用户留存的关键。 |
| M-03 | 岗位能力差距分析 | 系统提供目标岗位的能力模型拆解，用户可通过自评或在线测试，生成"当前能力 vs 岗位要求"的雷达图对比，标注差距等级和补全建议。 | 解决用户"不知道自己差多少"的焦虑，将模糊的焦虑转化为可执行的行动清单。 |
| M-04 | 简历智能解析与优化 | 用户上传简历和目标岗位JD，系统自动解析并生成匹配度评分，指出缺失的关键词/技能，基于STAR法则优化项目描述，提供简历模板。 | 求职期学生的刚需功能。调研显示简历过筛率低是求职最大瓶颈，此功能有极高的用户价值。 **2026-05: 画像解析已完成（POST /api/profile/resume），JD 诊断已完成（POST /api/jd/diagnose），简历优化尚在规划中** |
| M-05 | 技术八股/算法模拟面试 | 系统提供高频技术面试题库（覆盖Java、操作系统、网络、数据库、中间件等），支持AI追问式对话模拟面试，用户回答后AI给出评分和改进建议。 | 面试准备是求职期的核心场景。模拟面试功能将"被动背诵"转化为"主动输出+即时反馈"，学习效率提升数倍。 |
| M-06 | 用户画像与数据管理 | 系统记录用户的基本信息（学校、专业、年级、目标方向）、学习进度、求职进度，支持用户随时查看和更新个人档案。 | 所有个性化功能的基础支撑。没有用户数据，无法生成定制化路径和建议。 |

### Should Have（应该有） — 显著提升用户体验和价值的功能

| 需求编号 | 需求名称 | 需求描述 | 优先级理由 |
|----------|----------|----------|------------|
| S-01 | 项目实践指导与代码Review | 系统根据用户目标方向推荐差异化项目，提供需求文档和技术方案。用户实现后可提交代码，AI进行Review并给出架构、性能、代码规范方面的建议。 | 解决"项目同质化、缺乏深度"痛点。但技术实现复杂度高（代码Review准确性），初期可用简化版本替代。 |
| S-02 | 求职进度管理看板 | 提供可视化的求职全流程管理（投递→笔试→一面→二面→HR面→OC），支持设置提醒、记录面试反馈、生成数据分析报告（各环节转化率）。 | 解决求职进度混乱的痛点。但用户可在MVP阶段用外部工具替代，非绝对刚需。 |
| S-03 | 行业/岗位实时数据看板 | 整合互联网招聘数据（岗位数量、薪资分布、技能要求变化趋势），按方向（后端/前端/算法/测试）和城市展示市场热度。 | 帮助学生做出数据驱动的方向选择，降低信息不对称。但数据获取和更新需要持续投入。 |
| S-04 | 自我评估测评系统 | 提供融合霍兰德、技术偏好、学习风格的综合测评，生成"个人-方向匹配雷达图"和Top3推荐方向。 | 入学适应期学生的核心需求，帮助方向选择。但测评体系设计和验证需要一定周期。 |
| S-05 | Offer决策分析器 | 用户录入多个offer的关键信息，系统从7个维度进行量化评分，结合行业数据给出长期发展预测和决策建议。 | 解决offer选择期的决策困难。但用户使用频率低（一年最多1-2次），且决策辅助的准确性要求高。 |
| S-06 | 学习进度追踪与动态调优 | 用户标记学习完成情况，系统追踪整体进度，当进度落后时自动调整后续计划，或当发现用户某知识点薄弱时推荐补充材料。 | 将静态路径升级为动态自适应路径，显著提升学习效果。但需要较复杂的用户行为数据分析。 |

### Could Have（可以有） — 增强竞争力、提升用户粘性的功能

| 需求编号 | 需求名称 | 需求描述 | 优先级理由 |
|----------|----------|----------|------------|
| C-01 | 社区/同辈交流模块 | 用户可按方向/学校/目标公司建立小组，分享面经、交流进度、组队刷题。 | 同辈支持对职业规划有正向作用，但社区运营成本高，冷启动困难。 |
| C-02 | 导师/学长1v1对接预约 | 平台邀请大厂在职工程师/已毕业的学长入驻，用户可预约付费/免费的1v1咨询。 | 高价值但非标服务，质量控制和供需匹配难度大。 |
| C-03 | 校招时间节点日历 | 整合各大厂校招时间线（提前批、正式批、补录），支持订阅提醒。 | 信息整合价值明确，但数据维护成本高，且竞品（牛客、OfferShow）已有成熟产品。 |
| C-04 | 职业规划报告导出 | 用户可将AI生成的职业规划、能力评估、求职进度等导出为PDF/Word报告，用于与导师/家长沟通或存档。 | 提升用户感知价值，但技术实现简单，优先级不高。 |
| C-05 | 企业真题/面经库 | 按公司/岗位/年份结构化整理面试真题和面经，支持AI基于面经生成模拟面试题。 | 牛客网等已有大量UGC面经，差异化需要AI重新组织。 |
| C-06 | 多模态交互（语音/图片） | 支持语音对话咨询、上传手写笔记/书籍页面进行答疑、拍照上传简历进行解析。 | 提升交互便利性，但技术实现复杂，且文字交互已能满足核心需求。 |

### Won't Have（暂不实现） — 当前版本明确排除的功能

| 需求编号 | 需求名称 | 需求描述 | 暂不实现理由 |
|----------|----------|----------|--------------|
| W-01 | 直接内推/投递功能 | 用户通过平台直接向合作企业投递简历或获取内推码。 | 需要建立企业合作关系，涉及商务拓展、HR系统集成、数据隐私合规等复杂问题，远超当前产品阶段能力。 |
| W-02 | 在线课程/培训售卖 | 平台自营或代理售卖在线编程课程、培训服务。 | 业务重心偏移为教育电商平台，与AI职业规划智能体的产品定位不符；且课程质量控制和售后成本高。 |
| W-03 | 企业端招聘服务 | 为企业提供人才筛选、简历推荐、校园招聘SaaS服务。 | 从C端产品转向B端，商业模式、团队能力要求完全不同。 |
| W-04 | 跨专业职业规划（非计算机） | 为金融、法律、医学等其他专业学生提供职业规划服务。 | 需构建完全不同的知识库和能力模型，当前应聚焦计算机领域做深做透。 |
| W-05 | 线下活动/训练营组织 | 组织线下刷题营、面试训练营、企业参访等活动。 | 重运营、重本地化、边际成本高，当前阶段应优先打磨线上产品。 |
| W-06 | 薪资谈判/法务咨询 | 为用户提供offer薪资谈判策略、劳动合同法务咨询。 | 涉及法律合规风险，且使用场景低频，可推荐外部专业服务替代。 |

## 四、非功能需求

### 4.1 性能要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **响应时间** | 常规对话响应 ≤ 2秒；复杂路径生成（如学习路径规划）≤ 8秒 | 用户与AI智能体对话时，2秒内响应是良好体验的上限；超过5秒会产生明显等待焦虑 |
| **并发支持** | 支持至少 5000 用户同时在线，高峰期（如秋招季晚8-11点）不降级 | 秋招季用户需求集中，系统需能承受脉冲式流量 |
| **吞吐量** | 简历解析模块支持单用户5秒内完成解析；测评模块支持1000人/小时同时测评 | 简历解析涉及OCR/NLP，需优化为异步处理+进度通知 |
| **可用性** | 系统可用率 ≥ 99.5%（月度），计划维护窗口提前公告 | 求职期用户依赖度高，系统宕机可能导致错过关键时间节点 |
| **数据容量** | 支持百万级用户档案存储，单用户学习/求职数据保留至少3年 | 满足用户从入学到毕业后早期的全周期数据沉淀 |

### 4.2 可用性要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **多端适配** | 支持Web端（PC）、移动端H5、微信小程序三端，核心功能在移动端完整可用 | 学生群体移动端使用时长远高于PC，求职场景（如面试前临时复习）高度依赖手机 |
| **交互设计** | 对话界面采用"聊天式"设计，降低使用门槛；复杂输出（如学习路径、能力雷达图）采用可视化卡片 | 避免传统表单填写式的复杂交互，让用户"像聊天一样做规划" |
| **无障碍支持** | 核心功能支持屏幕阅读器，文字对比度符合WCAG 2.1 AA标准 | 确保视障学生群体的可访问性 |
| **新手引导** | 首次使用提供3-5步引导，帮助用户完成基础信息录入和首次对话 | 降低新用户流失率，快速建立"这产品能帮我"的认知 |
| **容错设计** | 用户输入模糊/错误时，AI智能体主动澄清（如"你是指Java后端开发还是Android开发？"）而非直接报错 | 学生用户表达能力参差不齐，系统需具备"引导式纠错"能力 |
| **离线缓存** | 已生成的学习路径、简历模板、测评报告支持本地缓存，弱网环境下可查看 | 学生群体在校园网/公共场所网络不稳定场景常见 |

### 4.3 安全性要求（涉及学生隐私数据）

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **数据分级** | 用户数据分为三级：公开级（昵称、学校）、敏感级（手机号、邮箱）、机密级（简历内容、测评结果、求职进度） | 简历中可能包含个人项目经历、联系方式等高度敏感信息 |
| **加密存储** | 敏感级和机密级数据采用AES-256加密存储；传输过程全链路TLS 1.3加密 | 防止数据库泄露导致的大规模隐私暴露 |
| **访问控制** | 用户数据严格隔离，任何管理员/运营人员查看用户原始数据需经审批留痕；AI模型训练使用脱敏数据 | 避免内部人员滥用数据，符合《个人信息保护法》要求 |
| **最小授权** | 第三方组件/服务（如云存储、AI API）仅授予最小必要权限，定期审计API调用日志 | 减少供应链攻击面 |
| **数据删除** | 用户可随时导出并删除全部个人数据，删除操作在72小时内物理清除（含备份） | 满足用户"被遗忘权"，符合GDPR/个保法合规要求 |
| **未成年人保护** | 对未满18岁用户（如大一新生中部分未成年），默认开启最高隐私保护模式，对话记录不上传至模型训练 | 符合未成年人网络保护相关法规 |
| **安全审计** | 每季度进行一次第三方安全渗透测试，漏洞修复SLA：高危24小时、中危7天、低危30天 | 建立持续安全运营机制 |

### 4.4 扩展性要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **模块化架构** | 系统采用微服务架构，核心模块（对话引擎、路径规划、简历解析、测评系统）独立部署、独立扩展 | 便于后续迭代和故障隔离 |
| **模型可替换** | AI对话引擎层与底层LLM解耦，支持在不同模型（GPT-4/Claude/国产大模型）之间切换，模型切换不改动业务逻辑 | 平衡成本、性能和合规性（如国内部署需用国产模型） |
| **插件化扩展** | 学习路径中的"学习资源推荐"、测评题库支持插件化接入，新资源/题库可通过配置化方式添加 | 便于对接第三方课程平台、快速扩充题库 |
| **多租户支持** | 数据模型支持未来向高校/培训机构输出SaaS版本（如某大学购买服务给学生使用），租户间数据完全隔离 | 为未来B端扩展预留架构能力 |
| **多语言预留** | 核心数据模型预留多语言字段，为后续面向港澳台/海外计算机留学生扩展做准备 | 全球化扩展的技术预埋 |
| **数据接口开放** | 提供标准REST API和Webhook，支持未来与高校教务系统、第三方招聘平台的数据打通 | 建立生态连接能力 |

## 五、需求优先级总览

### 5.1 MVP 版本（第一阶段：0-3个月）

**目标**：验证核心用户价值，覆盖"最痛的痛点"

功能范围：M-01 ~ M-06（全部Must Have）

**关键成功指标**：
- 用户首次对话完成率 ≥ 70%
- 学习路径生成后7日内用户回访率 ≥ 40%
- 简历优化功能使用率（上传简历用户数/总注册用户）≥ 25%

### 5.2 增强版本（第二阶段：4-6个月）

**目标**：完善核心场景闭环，提升用户粘性

功能范围：S-01 ~ S-06（全部Should Have）+ C-03（校招日历）+ C-04（报告导出）

**关键成功指标**：
- 用户周活跃率 ≥ 35%
- 模拟面试功能人均使用次数 ≥ 3次/月
- 求职进度管理功能在秋招季渗透率 ≥ 30%

### 5.3 生态版本（第三阶段：7-12个月）

**目标**：构建用户社区和生态连接

功能范围：C-01 ~ C-02（社区/导师对接）+ C-05（面经库）+ C-06（多模态）+ 多租户SaaS支持

**关键成功指标**：
- 社区功能日活跃用户占比 ≥ 15%
- 导师对接服务GMV（如有）达到自给自足
- 高校合作SaaS版本签约 ≥ 5 家

## 六、关键假设与风险

### 6.1 关键假设

1. **用户付费意愿假设**：学生群体付费能力有限，MVP阶段应以免费增值模式为主，高级功能（如1v1模拟面试、深度测评）可设置付费门槛
2. **AI回答质量假设**：大模型生成的职业规划建议需要达到"比学长学姐更系统、比辅导员更懂技术"的质量门槛，否则用户不会持续使用
3. **数据获取假设**：能够合法获取并持续更新互联网招聘数据、面经数据、公司信息，用于市场分析和岗位匹配

### 6.2 主要风险

| 风险 | 影响等级 | 缓解策略 |
|------|----------|----------|
| **AI幻觉（Hallucination）**：模型生成错误的技术信息或过时数据 | 高 | 建立技术知识库+RAG检索增强，关键数据人工审核；对AI输出添加"仅供参考，请交叉验证"提示 |
| **隐私泄露**：用户简历、测评数据被非法获取 | 高 | 严格的数据加密和访问控制；定期安全审计；制定应急响应预案 |
| **用户留存不足**：工具用完即走（如简历优化完就不再来） | 中 | 构建学习路径的持续追踪和动态调优，创造"陪伴式"使用场景 |
| **竞品压力**：牛客、LeetCode、脉脉等大平台快速跟进AI功能 | 中 | 聚焦"全链路职业规划"差异化定位，而非单点工具；深耕个性化和情感陪伴 |
| **合规风险**：AI生成内容涉及就业歧视建议或虚假承诺 | 中 | 建立内容审核机制；AI输出添加免责声明；避免给出绝对化承诺（如"保证进大厂"） |
